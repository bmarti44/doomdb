## Route and fixed boundary

`T4.3-EVAL | Terra | high | attempt 1`

The plan labels implementation Terra medium; evaluator authorship is elevated to
Terra high because the independent binary decoder, PNG parser, SQL/RLE
cross-representation checks, anti-cheat boundary, and mutation design are the
high-risk acceptance mechanism.

The candidate consumes the pending T4.2 public table SQL macro
`DOOM_R1_PIXELS(p_session VARCHAR2)` exactly as proposed: 64,000 rows keyed by
`(column_no,row_no)`, carrying `palette_index` and `layer_ordinal`. It does not
modify T4.1 or T4.2. Canonical frame order is column then row and the canonical
frame identity hashes the 64,000 raw palette bytes.

The three fixed poses are spawn east `(-416,256,0)`, spawn north
`(-416,256,90)`, and spawn south `(-416,256,270)`, all at eye Z 41. Spawn south
replaces the earlier central-west diagnostic because live upstream Oracle
validation found the latter's nearest solid is a closed two-sided line whose
facing upper, middle, and lower texture names are all `-`; accepting that pose
in T4.2 would require inventing a forbidden missing-texel fallback.

## Independent diagnostic path

The evaluator-only SQL adapter separately spools:

1. all ordered macro pixels;
2. the 256 PLAYPAL RGB entries from `DOOM_PALETTE_TEXEL`; and
3. analytic SQL RLE runs formed with `LAG` and a running change group.

The Node decoder imports no production parser or renderer. It verifies dense
coordinates and palette range, expands Appendix E runs, creates the 256,000-byte
raw RGBA frame, writes a deterministic 8-bit indexed PNG, and independently
parses PNG signature, IHDR, PLTE, IDAT, IEND, CRCs, filters, palette, and pixels.
Only those four chunks are accepted, so timestamps and other nondeterministic
metadata fail. Six full-column and eight pixel/RGBA diagnostics per pose are
written beside each PNG.

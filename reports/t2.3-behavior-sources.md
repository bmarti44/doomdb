# T2.3 behavior and source provenance

The E1M1 engine registry is independently and hand-authored for DoomDB. It is
not an engine port. Registry identities and behavior labels express the public,
observable game rules required by `PLAN.md`; state identifiers, transition
graphs, actions, RNG bytes, and JSON layout are project-owned designs.

## Inputs and licenses

- Freedoom Phase 1 0.13.0 supplies map placements and the art, music, sound,
  sprite, flat, texture, and patch bytes. These assets are BSD-3-Clause. The
  pinned archive and notices are under `vendor/freedoom/0.13.0/`.
- DoomWiki's WAD and gameplay documentation supplies public file-format and
  observable-behavior research under CC BY-SA 4.0.
- `PLAN.md`, under the repository's MIT license, supplies the project-specific
  scope, determinism, special semantics, and implementation constraints.

GPL Doom source is not an origin for any registry row, state graph, RNG value,
array, implementation code, or control flow. Public source implementations may
be consulted later only to audit externally observable behavior, with any such
audit recorded separately.

## Authorship decisions

The thing registry is closed over the 49 literal THING types parsed from the
pinned E1M1 marker range. Spawn markers deliberately have no visual state;
every other placed type roots a finite state graph. Monster graphs explicitly
name perception, chase, attack, pain, death, drops, sprite frames, and sounds.
Weapon graphs explicitly name ready, fire, refire, and flash states. Pickup
effects use bounded semantic identifiers so later SQL can implement limits as
data rather than infer them from display names.

The asset manifest is derived from the production parser's E1M1 sidedefs and
sectors, then closed over animation frames, texture PNAMES dependencies,
reachable state sprites and sounds, and required UI/audio roots. Each row has a
stable `(kind,name)` identity, sorted reasons, and literal source lump names.

Gameplay randomness is the first 256 bytes obtained by concatenating SHA-256
digests of ASCII labels `DoomDB project RNG v1|0000`, `...|0001`, and onward.
Runtime reads the checked-in table, returns the current byte, and persists the
cursor advanced modulo 256 exactly once per gameplay random read.
